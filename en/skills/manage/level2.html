Level definition

Level 2 of the MANAGE competency corresponds to complex data management. This competency represents the second level of learning (Critical Learning AC24). At this level, I am able to optimize query performance (improve execution speed), use analysis tools like Power BI (data visualization tool), and manage large volumes of data (large amounts of data).

This level represents an evolution in my ability to analyze and visualize data, optimize performance, and manage larger volumes of data. These skills were developed during semesters 3 and 4 (second year of BUT Computer Science) through educational resources R3.01, R3.07, R3.08, R3.09, R3.10, R3.11, R4.03, R4.06, R4.10 and R4.11, as well as academic projects SAÉ 2.04, SAÉ 3.01, SAÉ 4.01 and my professional internship at Helvetics Events.

Traces and contextualized evidence

Here are my traces and evidence for Level 2:

EVIDENCE 1: Clone Uber (Laravel) - Complex database

Context: Academic project carried out in the second year (Semester 3) as part of SAÉ 3.01. This involved developing a complete web application similar to Uber using Laravel, requiring management of a complex database with performance optimization and data security.

Trace:

  • Optimized relational database with indexing
  • Optimized SQL queries for performance
  • Data security and integrity implemented
  • Source code with data management available on GitHub
  • Detailed project sheet (see project sheet)

Justification: Thanks to this project, I learned to optimize database performance (AC24.01) by using indexing and optimizing my SQL queries to make them faster. I also discovered the importance of securing and ensuring data integrity (AC24.02) by implementing authentication, authorization and constraints to protect data. I applied knowledge from resources R2.06 (Database exploitation) and R3.07 (Triggers, Procedures).

Reflection: This project was a major challenge in terms of data management as it required managing large amounts of data (users, trips, calculations) while ensuring performance and security. The main difficulty was optimizing SQL queries to guarantee acceptable response times. I overcame this difficulty by analyzing query execution plans, using appropriate indexing, and applying techniques taught in resources R2.06 and R3.07. This project showed me the importance of optimization and security in production data management.

EVIDENCE 2: Clone Uber (.NET + Vue.js) - PostgreSQL with Code First

Context: Academic project carried out in the second year (Semester 4) as part of SAÉ 4.01. This involved developing a complete web application similar to Uber using .NET Core for the backend and Vue.js for the frontend, with complex data management using PostgreSQL with Code First.

Trace:

  • Optimized relational database (PostgreSQL) with Code First
  • Data modeling with Entity Framework Core
  • Database performance optimization
  • Data security and integrity implemented
  • Source code with data management available on GitHub
  • Detailed project sheet (see project sheet)

Justification: This project allowed me to learn to optimize relational database performance (AC24.01) by using PostgreSQL with Code First and Entity Framework Core. I discovered how to model data with Code First and how to optimize queries generated by the ORM. I also learned to ensure data security and integrity (AC24.02) in a .NET Core application. I used resources R2.06 (Database exploitation) and R3.07 (Stored procedures).

Reflection: This project was enriching as it allowed me to work with Entity Framework Core and Code First, a modern approach to data modeling. The main difficulty was understanding how to optimize performance with an ORM and how to properly structure data models. I overcame this difficulty by studying Entity Framework Core best practices, optimizing LINQ queries, and applying concepts taught in resources R2.06 and R3.07. This experience showed me the importance of understanding the ORM well to optimize database performance.

EVIDENCE 3: MongoDB - NoSQL database

Context: Academic project carried out in the second year (Semester 4) as part of resource R4.03 (Quality and beyond relational - MongoDB). This involved mastering MongoDB, a NoSQL database, through practical exercises covering simple queries, aggregation framework, updates, modeling, transactions, JSON Schema validation, indexing and performance analysis.

Trace:

  • GitHub repository with complete MongoDB exercises: https://github.com/oeoecbien/mongodb
  • Simple queries with comparators, logical operators, projection, regex, sorting and pagination
  • Aggregation framework with stages $match, $project, $unwind, $group, $sort, $lookup
  • Updates with updateOne, updateMany, operators $set, $unset
  • Modeling with collection duplication, bulk insertion, simple and enriched references
  • Transactions with sessions, startTransaction, commitTransaction, abortTransaction
  • JSON Schema validation with constraints required, enum, minimum, maximum, maxItems
  • Indexing with different types of indexes (simple, compound, text, multikey, unique, TTL, partial, sparse, collation)
  • Performance analysis with explain("executionStats")

Justification: This project allowed me to learn to optimize NoSQL database performance (AC24.01) by mastering MongoDB and its indexing and performance analysis features. I also discovered how to ensure data coherence and quality (AC24.02) in a NoSQL database by using transactions and JSON Schema validation. I used resource R4.03 (MongoDB) to develop comprehensive expertise in NoSQL databases.

Reflection: This project was fundamental for mastering MongoDB and understanding the differences between relational and NoSQL databases. The main difficulty was understanding MongoDB-specific concepts like the aggregation framework, transactions in a NoSQL context, and multi-dimensional indexing. I overcame this difficulty by practicing regularly with MongoDB, studying the official documentation, and applying concepts taught in resource R4.03. This experience showed me the importance of choosing the right type of database according to project needs and allowed me to develop expertise in NoSQL databases.

Pedagogical reflection and progression
Challenges encountered
The main challenges at this level concerned SQL query performance optimization, NoSQL database management (MongoDB), and understanding the differences between relational and NoSQL databases.
Solutions provided
To overcome these difficulties, I studied SQL optimization techniques, practiced intensively with MongoDB through comprehensive exercises, worked on full-stack projects using different types of databases, and applied methods taught in class.

Progression Level 2 allowed me to develop my complex data management skills. Through projects like Clone Uber (SAÉ 3.01 and SAÉ 4.01) and MongoDB (R4.03), I learned to optimize performance of both relational and NoSQL databases, ensure data security and integrity, and master different types of databases. These experiences allowed me to develop expertise in complex data management, from performance optimization to NoSQL database mastery.
